Mari kita bahas salah satu dari tiga pilar utama Pemrograman Berorientasi Objek (OOP). Jika _Class_ dan _Object_ adalah cara kita mencetak data, maka materi kali ini adalah cara kita **melindungi** data tersebut dari tangan-tangan jahil, baik dari pengguna aplikasi maupun dari _programmer_ lain di tim Anda.

---

## 1. Definisi Singkat

**Enkapsulasi (Encapsulation)** adalah konsep menyembunyikan detail internal dari sebuah objek dan hanya memperlihatkan apa yang benar-benar perlu dilihat oleh dunia luar. Untuk mengatur siapa yang boleh melihat dan mengubah data ini, OOP menggunakan hak akses (_Access Modifiers_):

- **Public:** Terbuka untuk umum. Bisa diakses dan diubah dari mana saja.
    
- **Private:** Sangat rahasia. Hanya bisa diakses dan diubah dari **dalam** _class_ itu sendiri.
    
- **Protected:** Rahasia keluarga. Hanya bisa diakses dari dalam _class_ itu sendiri dan _class_ turunannya (anaknya).
    

## 2. Analogi: Mesin ATM

Bayangkan sebuah **Mesin ATM** di bank:

- **Public (Layar, Tombol Angka, dan Lubang Kartu):** Ini adalah bagian yang sengaja disediakan untuk berinteraksi dengan Anda sebagai nasabah. Anda bebas menekan tombolnya.
    
- **Private (Brankas Uang di Dalam):** Kotak besi berisi uang tunai di dalam mesin. Anda (sebagai pihak luar) **TIDAK BOLEH** membuka brankas ini dan mengambil uangnya secara langsung.
    
- **Bagaimana cara ambil uangnya?** Anda menekan tombol _Public_ ("Tarik Tunai 50 Ribu"), lalu sistem komputer di dalam ATM (Metode _Private_) akan mengecek saldo Anda. Jika cukup, sistem internal yang akan mengambil uang dari brankas _Private_ dan mengeluarkannya lewat lubang uang. Anda hanya berinteraksi dengan "jalur resmi", bukan mencuri langsung dari brankasnya.
    

## 3. Poin-Poin Penting

- **Mencegah Kerusakan Data:** Bayangkan jika saldo rekening bank (atribut) bersifat _Public_. Siapapun bisa mengetik `rekeningBudi.saldo = 1000000000` di konsol _browser_. Dengan membuatnya _Private_, kita memaksa perubahan saldo harus melewati metode resmi yang punya aturan ketat (misal: `tambahSaldo(jumlah)`).
    
- **Getters dan Setters:** Ini adalah dua istilah wajib dalam Enkapsulasi.
    
    - **Getter** adalah metode _Public_ untuk **membaca** nilai _Private_ (hanya _read-only_).
        
    - **Setter** adalah metode _Public_ untuk **mengubah** nilai _Private_ (dengan syarat/validasi).
        
- **Di JavaScript:** Dulunya JavaScript tidak punya fitur _Private_ yang sesungguhnya (programmer biasanya hanya menaruh tanda _underscore_ seperti `_saldo` sebagai kode etik "jangan disentuh"). Namun di JavaScript modern, kita menggunakan simbol _hashtag_ (`#`) untuk membuat atribut benar-benar _Private_.
    

## 4. Contoh Implementasi

Mari kita buat sistem dompet digital menggunakan **JavaScript** modern:

JavaScript

```
class DompetDigital {
    // 1. PRIVATE ATTRIBUTE (Ditandai dengan '#')
    // Uang di dalam dompet ini tidak bisa disentuh dari luar
    #saldo; 

    constructor(namaPemilik, saldoAwal) {
        this.pemilik = namaPemilik; // Ini PUBLIC (bisa dilihat siapa saja)
        this.#saldo = saldoAwal;    // Ini PRIVATE (tersembunyi)
    }

    // 2. GETTER: Jalur resmi untuk "Melihat" saldo
    cekSaldo() {
        return `Saldo ${this.pemilik} saat ini: Rp${this.#saldo}`;
    }

    // 3. SETTER: Jalur resmi untuk "Mengubah" saldo (Top Up)
    // Di sini kita bisa pasang penjaga/validasi!
    topUp(jumlah) {
        if (jumlah <= 0) {
            console.log("Top Up gagal: Jumlah harus lebih dari 0!");
        } else {
            this.#saldo = this.#saldo + jumlah;
            console.log(`Top Up berhasil! Saldo bertambah Rp${jumlah}`);
        }
    }
}

let dompetBudi = new DompetDigital("Budi", 50000);

// --- MARI KITA UJI KEAMANANNYA ---

// Mengakses atribut Public? BISA!
console.log(dompetBudi.pemilik); // Output: "Budi"

// Mencoba membobol atribut Private langsung dari luar? GAGAL!
// console.log(dompetBudi.#saldo); // Akan menghasilkan ERROR!
// dompetBudi.#saldo = 10000000;   // Hacker mencoba ubah saldo? ERROR!

// Cara yang benar: Lewat "Jalur Resmi" (Getter dan Setter)
console.log(dompetBudi.cekSaldo()); // Output: Saldo Budi saat ini: Rp50000

dompetBudi.topUp(-1000); // Ditolak oleh sistem validasi kita
dompetBudi.topUp(20000); // Berhasil
console.log(dompetBudi.cekSaldo()); // Output: Saldo Budi saat ini: Rp70000
```

## 5. Koneksi dengan Web Development

Dalam **Web Development**, Enkapsulasi adalah pahlawan keamanan aplikasi Anda:

1. **Menyembunyikan API Key (Kunci Rahasia):** Saat aplikasi Anda berkomunikasi dengan layanan luar (seperti _server_ pembayaran Stripe atau Midtrans), aplikasi butuh _Secret Key_. Jika _key_ ini diletakkan di atribut _Public_, _hacker_ bisa membacanya dan menguras uang perusahaan. _Developer_ mengenkapsulasinya di dalam sistem _Backend_ yang tertutup rapat.
    
2. **State Management (React/Redux):** Di React, data aplikasi (_State_) sangat dijaga ketat. Komponen web tidak bisa sembarangan mengubah _State_ milik komponen lain secara langsung. Mereka harus mengirim pesan (_Action_) agar sistem internal yang mengubahnya. Ini persis seperti konsep _Setter_.
    
3. **Data Validation di Form:** Saat pengguna mengisi formulir pendaftaran, umur tidak boleh negatif atau teks tidak boleh diisi kode berbahaya (SQL Injection). _Developer_ memasukkan proses pembersihan data ini di dalam _Setter_ objek `User` sebelum datanya dikirim ke _database_.

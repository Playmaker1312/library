## 1. Definisi Singkat

**Analisis Algoritma** adalah proses mengevaluasi seberapa efisien sebuah blok kode (algoritma) dalam menyelesaikan masalah, terutama ketika jumlah data yang harus diproses semakin membesar. Evaluasi ini difokuskan pada dua metrik utama:

1. **Time Complexity (Kompleksitas Waktu):** Seberapa lama waktu eksekusi program bertambah seiring bertambahnya jumlah data.
    
2. **Space Complexity (Kompleksitas Ruang/Memori):** Seberapa banyak kapasitas memori (RAM) yang disedot oleh program seiring bertambahnya jumlah data.
    

## 2. Analogi: Mencari Buku di Perpustakaan Raksasa

Bayangkan Anda ditugaskan mencari buku "Harry Potter" di sebuah perpustakaan:

- **Algoritma A (Mengecek Satu per Satu):** Anda berjalan dari rak pertama, membaca judul buku satu per satu sampai menemukan buku tersebut.
    
- **Algoritma B (Menggunakan Komputer Indeks):** Anda mengetik judul di komputer perpustakaan, komputer langsung memberi tahu: "Rak B, Baris 3". Anda langsung berjalan ke sana dan mengambilnya.
    

Jika perpustakaan itu **hanya punya 10 buku**, Algoritma A dan B akan terasa sama cepatnya (mungkin selisih sepersekian detik).

Tetapi, bagaimana jika perpustakaan itu adalah Perpustakaan Nasional yang memiliki **10 juta buku**? Algoritma A akan memakan waktu berbulan-bulan, sementara Algoritma B tetap hanya memakan waktu beberapa detik. Inilah mengapa kita harus menganalisis algoritma!

## 3. Poin-Poin Penting

- **Fokus pada Skenario Terburuk (Worst-Case Scenario):** Saat menganalisis kode, _programmer_ selalu pesimis. Kita tidak menghitung seberapa cepat kode berjalan jika datanya kebetulan ada di urutan pertama, melainkan seberapa lambat kodenya jika datanya ada di urutan paling akhir (atau tidak ada sama sekali).
    
- **Big O Notation:** Ini adalah "bahasa gaul" universal para _programmer_ untuk menyebutkan efisiensi algoritma. Anda akan sering mendengar istilah seperti **O(1)** (Kecepatan kilat/konstan), **O(n)** (Kecepatan sebanding dengan jumlah data), atau **O(n²)** (Sangat lambat jika datanya banyak).
    
- **Hardware Bukan Solusi Utama:** Komputer seharga Rp 100 juta yang menjalankan algoritma buruk (O(n²)) akan kalah telak oleh laptop murah seharga Rp 3 juta yang menjalankan algoritma efisien (O(1)) saat memproses jutaan data.
    

## 4. Contoh Implementasi

Mari kita lihat bukti nyatanya di **JavaScript**. Kita punya satu tugas sederhana: **Menjumlahkan semua angka dari 1 sampai N**.

JavaScript

```
// ==========================================
// ALGORITMA A: Menggunakan Perulangan (Looping)
// ==========================================
function jumlahkanDenganLoop(n) {
    let total = 0;
    // Komputer harus bekerja melakukan penambahan sebanyak 'n' kali
    for (let i = 1; i <= n; i++) {
        total = total + i; 
    }
    return total;
}

// ==========================================
// ALGORITMA B: Menggunakan Rumus Matematika (Gauss)
// ==========================================
function jumlahkanDenganRumus(n) {
    // Komputer HANYA melakukan 3 operasi matematis, berapapun besarnya 'n'
    return n * (n + 1) / 2;
}

// --- MARI KITA UJI EFISIENSINYA ---

// Jika n = 10, keduanya terasa instan.
console.log(jumlahkanDenganLoop(10));  // Komputer mengulang 10 kali
console.log(jumlahkanDenganRumus(10)); // Komputer menghitung 1 kali

// Tapi jika n = 1.000.000.000 (Satu Miliar)
console.log(jumlahkanDenganLoop(1000000000));  
// Algoritma A akan membuat komputer Anda "nge-freeze" beberapa detik karena harus mengulang 1 miliar kali!

console.log(jumlahkanDenganRumus(1000000000)); 
// Algoritma B akan langsung mengeluarkan hasil dalam 0.0001 detik. Sangat efisien!
```

## 5. Koneksi dengan Web Development

Analisis algoritma bukan sekadar teori anak kuliahan, ini adalah masalah hidup dan mati bagi aplikasi web Anda:

1. **Loading Data (Frontend):** Jika Anda membuat web _e-commerce_ dan mencoba me-_render_ 10.000 data produk menggunakan algoritma perulangan ganda (O(n²)) untuk mencari produk duplikat, _browser_ pengguna akan membeku (_Not Responding_) dan mereka akan menutup _website_ Anda.
    
2. **Database Queries (Backend):** Saat _backend developer_ salah memilih struktur data (misal: mencari nama di dalam Array raksasa alih-alih menggunakan Dictionary/Hash Map), pencarian _user_ yang seharusnya instan bisa memakan waktu 5-10 detik. Ini sangat membebani _server_ (_Server Overload_).
    
3. **Optimasi Algoritma Pencarian:** Fitur _Search Bar_ yang bisa mengetik dan langsung memunculkan hasil tanpa perlu menekan _Enter_ (_Autocomplete_) sangat mengandalkan struktur data dan analisis algoritma tingkat tinggi (seperti struktur _Trie_ atau _Debouncing_) agar tidak membuat _server crash_ karena dihujani _request_.

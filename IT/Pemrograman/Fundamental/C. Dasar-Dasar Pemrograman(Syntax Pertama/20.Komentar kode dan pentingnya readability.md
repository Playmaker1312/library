## 1. Definisi Singkat

**Komentar Kode (Code Comments)** adalah teks penjelasan di dalam _source code_ yang sepenuhnya diabaikan oleh komputer saat program dijalankan; ia ditulis khusus untuk dibaca oleh manusia. **Readability (Keterbacaan)** adalah tolak ukur seberapa mudah, rapi, dan cepat struktur kode tersebut dipahami oleh _programmer_ lain (maupun diri Anda sendiri di masa depan).

## 2. Analogi: Catatan Pinggir pada Resep Keluarga

Bayangkan Anda memiliki buku resep rahasia keluarga turun-temurun.

- **Kode** adalah langkah-langkah mutlaknya (misal: "Masukkan 2 sendok takar garam").
    
- **Komentar** adalah catatan kecil tulisan tangan kakek Anda di pinggir halaman ("_Gunakan garam laut merk X, karena rasanya tidak terlalu asin_").
    
- **Readability** adalah seberapa rapi tulisan tangan dan tata letak buku tersebut. Jika resepnya berantakan, tercoret-coret, dan tanpa spasi, Anda (sebagai koki penerus) mungkin akan kebingungan membedakan takaran gula dan garam saat mencoba memasaknya sepuluh tahun kemudian.
    

## 3. Poin-Poin Penting

- **Fungsi Utama Komentar:** Menjelaskan _mengapa_ (why) sebuah kode ditulis dengan cara tertentu, bukan _apa_ (what) yang dilakukan kode tersebut. Jika kode Anda ditulis dengan rapi, orang sudah tahu _apa_ fungsinya tanpa perlu dijelaskan.
    
- **Mencegah "Amnesia Kode":** Anda dijamin akan lupa bagaimana cara kerja kode kompleks yang Anda tulis sendiri 3 bulan yang lalu jika tidak ada catatan atau strukturnya berantakan.
    
- **Kolaborasi Tim:** Di dunia nyata, _software_ dibangun oleh puluhan orang. Kode yang _readable_ mengurangi waktu stres bagi _developer_ baru yang harus melanjutkan pekerjaan Anda.
    
- **Debugging Sementara (Commenting Out):** Komentar sering digunakan untuk "mematikan" baris kode tertentu sementara waktu saat sedang mencari _error_ (bug), agar Anda tidak perlu repot menghapus dan mengetik ulang kodenya.
    
- **Mitos Pemula:** _Programmer_ pemula sering berpikir "yang penting programnya jalan". _Programmer_ profesional tahu bahwa "kode dibaca oleh manusia sepuluh kali lebih sering daripada ditulis oleh mesin".
    

## 4. Contoh Implementasi

Di dalam **JavaScript**, kita menggunakan `//` untuk komentar satu baris, dan `/* ... */` untuk komentar banyak baris. Perhatikan perbedaan keterbacaan di bawah ini:

JavaScript

```
// ❌ CONTOH BURUK (Readability rendah, penamaan variabel tidak jelas, komentar tidak berguna)
let x = 10000; // bikin variabel x isi 10000
let y = x * 0.11; // kali dengan 0.11
let z = x + y; // tambah x dan y

// ✅ CONTOH BAIK (Readability tinggi, variabel jelas, komentar memberi konteks "Mengapa")
let hargaLaptop = 10000;

// Menggunakan PPN 11% sesuai dengan penyesuaian tarif pajak pemerintah tahun 2022
let nilaiPajak = hargaLaptop * 0.11; 
let totalHarga = hargaLaptop + nilaiPajak;
```

## 5. Koneksi dengan Web Development

Dalam **Web Development**, _readability_ dan komentar adalah instrumen wajib untuk bekerja dalam tim:

1. **Handoff (Serah Terima) Desain ke Kode:** Saat _Frontend Developer_ menyusun struktur kerangka halaman web (HTML), halamannya bisa mencapai ribuan baris. Mereka WAJIB menggunakan komentar untuk menandai batas antar bagian, misalnya: `dan` agar tidak tersesat di dalam kode sendiri.
    
2. **Dokumentasi API Publik:** Jika _Backend Developer_ membuat layanan yang akan dipakai oleh pihak luar (misalnya API untuk gerbang pembayaran), mereka menggunakan format komentar khusus untuk merancang dokumen pedoman (_documentation_).
    
3. **Standar Industri (Linter):** Tim _Web Developer_ modern menggunakan alat bantu (seperti ESLint atau Prettier) yang akan secara otomatis merapikan spasi, letak kurung kurawal, dan spasi kode Anda setiap kali Anda menekan tombol _Save_. Ini memaksa seluruh tim memiliki standar _readability_ yang sama persis.
